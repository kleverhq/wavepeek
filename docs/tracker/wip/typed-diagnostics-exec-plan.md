# Implement typed diagnostics and fatal errors for v1

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Note that this document must be maintained in accordance with the `exec-plan` skill. It is intentionally self-contained: a contributor should be able to start from a clean checkout of this branch, read only this plan, and implement the behavior end to end.

## Purpose / Big Picture

`wavepeek` is preparing a v1 major release. Today successful JSON output contains `warnings: ["..."]`, while process-level failures print `error: <category>: <message>` on stderr. After this change, successful command output will carry typed diagnostic objects in a required `diagnostics` array, and process-level failures will print `fatal: <category>: <message>` without any JSON envelope. This gives scripts and agents stable machine-readable codes for non-fatal conditions while keeping true command failures fail-fast and stderr-only.

A user will be able to run `wavepeek scope --waves tests/fixtures/hand/m2_core.vcd --filter DOES_NOT_MATCH --json` and see a successful JSON envelope with `data: []` and a `diagnostics` entry containing `kind: "warning"`, `code: "WPK-W0003"`, and a human-readable message. The same command without `--json` will leave stdout empty and print `warning[WPK-W0003]: ...` to stderr. A fatal invocation such as `wavepeek info --waves missing.vcd --json` will leave stdout empty and print `fatal: file: cannot open ...` to stderr with a non-zero exit code.

## Non-Goals

This plan does not add suppress, promote, or downgrade controls for diagnostics. It does not add a JSON error envelope, a JSON Lines streaming mode, compatibility aliases for the old `warnings` field, compatibility with the old `error:` fatal prefix, or extended diagnostic fields such as `details`, `span`, `help`, or `url`. It does not remove `schema/wavepeek_v0.json`; prior major schema artifacts are public contracts and must remain available.

## Surprises & Discoveries

- Observation: `wavepeek schema` currently prints the checked-in schema artifact through `include_str!` in `src/schema_contract.rs`; it does not synthesize JSON Schema from Rust types at runtime.
  Evidence: `src/schema_contract.rs` defines `CANONICAL_SCHEMA_JSON` with `include_str!(... "/schema/wavepeek_v" ... ".json")`, and `src/engine/schema.rs` returns that string directly.

- Observation: Fatal errors are generated by `WavepeekError` display strings, not by a separate renderer.
  Evidence: `src/error.rs` currently derives display strings such as `#[error("error: args: {0}")]`; `src/main.rs` prints `eprintln!("{error}")` on failure.

- Observation: Existing non-fatal warnings are plain strings produced in `scope`, `signal`, and `change` command engines, plus empty `warnings` vectors in other command engines.
  Evidence: `src/engine/scope.rs`, `src/engine/signal.rs`, and `src/engine/change.rs` push strings such as `limit disabled: --max=unlimited`, `limit disabled: --max-depth=unlimited`, `truncated output to ...`, and `no signal changes found in selected time range`.

- Observation: `just check` is the local pre-handoff gate but does not run the main integration test suite; `just test` runs `cargo test -q`, and `just ci` is the fuller CI-shaped gate.
  Evidence: `justfile` defines `test: require-container check-rtl-artifacts` with `cargo test -q`, `check: format-check lint check-schema check-actions check-bench-e2e-fsdb-catalog check-build docs-site-check check-commit`, and `ci: format-check lint check-schema check-actions test-aux coverage-src-check check-build docs-site-check`.

- Observation: The benchmark E2E harness stores normalized wavepeek functional artifacts with only `data` and `warnings`, not full envelopes, and 284 tracked `.wavepeek.json` baseline artifacts currently contain the old `warnings` key.
  Evidence: `bench/e2e/perf.py` validates and stores `payload["warnings"]`; `git ls-files 'bench/e2e/runs/*.wavepeek.json' 'bench/e2e/runs/**/*.wavepeek.json' | wc -l` returned `284`.

- Observation: The first Rust slice needed explicit `#[allow(dead_code)]` annotations for future-facing diagnostic pieces that v1 reserves but does not emit yet.
  Evidence: the pre-commit hook ran `cargo clippy --all-targets -- -D warnings` and rejected the unused `Info`/`Error` variants, `ErrorDiagnosticCode`, and `Diagnostic::info`/`Diagnostic::error` before those annotations were added.

- Observation: The root schema `data.oneOf` rejected valid empty successful results once multiple command payloads could be empty arrays.
  Evidence: code/schema review found that `[]` matches scope, signal, change, and property array definitions simultaneously; `schema/wavepeek_v1.json` now uses `data.anyOf` and command-specific `allOf` branches provide the narrowing.

- Observation: Benchmark functional artifact validation needed to validate typed diagnostic objects, not only the presence of a `diagnostics` list.
  Evidence: benchmark review found that `diagnostics: ["legacy warning"]` and artifacts retaining a stale `warnings` key would otherwise pass harness validation; `bench/e2e/perf.py` now rejects both.

- Observation: Diagnostic code validation also needs per-kind prefixes, not only the shared `WPK-[WE]####` shape.
  Evidence: the independent control review found that schema and benchmark validation would accept `kind: "warning"` with `WPK-E####` or `kind: "error"` with `WPK-W####`; both validators now require `WPK-W####` for warnings and `WPK-E####` for non-fatal errors.

## Decision Log

- Decision: Empty successful list/search-style results will emit `WPK-W0003`, not only `change` no-delta results.
  Rationale: A successful command with no stdout is ambiguous for agents and scripts. A diagnostic makes the empty result explicit without changing the process exit code.
  Date/Author: 2026-06-09 / user and planning agent.

- Decision: The JSON Schema will validate diagnostic codes with a pattern, not a closed enum.
  Rationale: A pattern such as `^WPK-[WE][0-9]{4}$` allows new warning or non-fatal error codes within the v1 line without editing clients that pin an older schema for the same major contract.
  Date/Author: 2026-06-09 / user and planning agent.

- Decision: Multiple diagnostics may share the same code in one result.
  Rationale: The code describes the diagnostic class, not a unique event. For example, disabling both `--max` and `--max-depth` should produce two `WPK-W0001` diagnostics with different messages.
  Date/Author: 2026-06-09 / user and planning agent.

- Decision: `wavepeek schema` remains a special helper command that prints a raw JSON Schema document and is not wrapped in the normal success envelope.
  Rationale: The schema command is the authority for the envelope contract and must not recursively wrap the schema artifact inside that same envelope.
  Date/Author: 2026-06-09 / user and planning agent.

- Decision: Keep `schema/wavepeek_v0.json` and other prior major schema files in the repository.
  Rationale: Major schema artifacts are public contracts and must remain available in the repository and documentation site after later majors ship.
  Date/Author: 2026-06-09 / user and planning agent.

- Decision: Reserve the non-fatal diagnostic error namespace `WPK-E####` now, even though the initial implementation only emits warning diagnostics.
  Rationale: The diagnostic model has `kind = "error"` for future partial-success scenarios; the schema and renderer should be ready for `error[WPK-E0001]: ...` without inventing another format later.
  Date/Author: 2026-06-09 / user and planning agent.

- Decision: Keep `WavepeekError::Unimplemented` as fatal category `unimplemented` if it is ever used.
  Rationale: It is already a distinct process-level category in the Rust error enum, and the v1 fatal contract can represent it as `fatal: unimplemented: ...` without remapping to `internal`.
  Date/Author: 2026-06-09 / user and planning agent.

- Decision: Mention the removal of the old `warnings` JSON field and old `error:` fatal prefix only once in `CHANGELOG.md`, not in public docs or command topics.
  Rationale: Public v1 docs should describe current stable behavior directly. The changelog is the correct place for migration history.
  Date/Author: 2026-06-09 / user and planning agent.

## Context and Orientation

The repository root is `/workspaces/wavepeek`. The project is a Rust CLI. Commands parse arguments in `src/cli/`, execute in `src/engine/`, render output in `src/output.rs`, and report process-level failures through `src/error.rs` and `src/main.rs`.

A process-level failure means the command did not return a successful command result. In this repository those failures are represented by `WavepeekError` in `src/error.rs`; `src/main.rs` prints the error to stderr and exits with the category's exit code. A non-fatal diagnostic means the command succeeded and returned data, but there is metadata about the result that automation should see, such as truncation or an empty result. Those diagnostics belong in `CommandResult` and in the JSON envelope.

The normal JSON envelope is rendered in `src/output.rs`. Today `OutputEnvelope<T>` has fields `$schema`, `command`, `data`, and `warnings`, and `render_json` builds the envelope from `CommandResult.warnings`. Human warnings are currently rendered by `emit_human_warnings`, which prints `warning: <message>` to stderr.

The command result type lives in `src/engine/mod.rs`. Today it has `command`, `json`, `human_options`, `data`, and `warnings: Vec<String>`. The `CommandData` enum in the same file describes the payload shapes. List-like or search-like successful payloads are `CommandData::Scope(Vec<ScopeEntry>)`, `CommandData::Signal(Vec<SignalEntry>)`, `CommandData::Change(Vec<ChangeSnapshot>)`, `CommandData::Property(Vec<PropertyCaptureRow>)`, `CommandData::DocsTopics(DocsTopicsData { topics })`, and `CommandData::DocsSearch(DocsSearchData { matches })`. `CommandData::Info` and `CommandData::Value` are not semantically empty when valid because they contain metadata or requested samples. `CommandData::Schema` is special because `wavepeek schema` prints a raw schema document instead of an envelope.

Current non-fatal warning producers are in `src/engine/scope.rs`, `src/engine/signal.rs`, and `src/engine/change.rs`. `scope` and `signal` warn when a user disables limits and when output is truncated. `change` warns when `--max=unlimited`, when output is truncated, and when no signal changes are found in the selected range. `property` currently returns empty rows without warning. `docs search` currently returns empty matches without warning. `scope` and `signal` can return empty arrays after filtering without warning.

The canonical current-major schema artifact is `schema/wavepeek_v1.json`. It is checked in and embedded into the binary by `src/schema_contract.rs`. `wavepeek schema` prints exactly those bytes. Because there is no Rust type-to-schema generator here, changing the machine contract requires deliberately editing `schema/wavepeek_v1.json` and then running `just check-schema` to prove that the runtime schema output matches the artifact. Do not edit or delete `schema/wavepeek_v0.json`.

Public docs under `docs/public/` are embedded into the `wavepeek docs` runtime. Stable output behavior is documented in `docs/public/reference/machine-output.md`; cross-command bounded-output semantics are documented in `docs/public/reference/command-model.md`; individual command examples live in `docs/public/commands/*.md`. Maintainer guidance that mentions output behavior appears in `docs/dev/style.md` and `docs/dev/architecture.md`. The packaged agent skill is `docs/skills/wavepeek.md`. The changelog is `CHANGELOG.md`.

Tests are split between Rust unit tests under `src/` and integration tests under `tests/`. Many integration tests currently assert `warnings` and `error:`. The benchmark E2E harness under `bench/e2e/` also validates normalized wavepeek artifacts; those artifacts are not full JSON envelopes, but they preserve the functional `data` plus the non-fatal metadata field, which must become `diagnostics`. Use `rg -n "warnings|warning:|error:" src tests docs schema tools bench CHANGELOG.md` during implementation to find contract updates. The standard pre-handoff gate is `just check`; the integration test gate is `just test`; the full CI-shaped gate is `just ci`.

## Plan of Work

Start by adding a typed diagnostic model. Create a new module `src/diagnostic.rs` and add `mod diagnostic;` to `src/lib.rs`. Define `DiagnosticKind` with serialized values `info`, `warning`, and `error`. Define a `Diagnostic` struct serialized as an object with `kind`, optional `code`, and `message`. The `code` field must use `#[serde(skip_serializing_if = "Option::is_none")]` so info diagnostics omit it. Keep constructors strict: `Diagnostic::info(message)` creates no code, `Diagnostic::warning(code, message)` requires a warning code, and a future-facing `Diagnostic::error(code, message)` requires an error code. Define `WarningDiagnosticCode` with initial variants `LimitDisabled`, `OutputTruncated`, and `EmptyResult`, returning `WPK-W0001`, `WPK-W0002`, and `WPK-W0003`. Define an `ErrorDiagnosticCode` enum or equivalent reserved type even if it has no emitted variants yet; if Rust requires at least one variant for convenient testing, use a private test-only helper instead of shipping a fake public code. Add methods needed by renderers and tests, such as `kind()`, `code()`, and `message()`.

Then change the command result plumbing. In `src/engine/mod.rs`, import `crate::diagnostic::Diagnostic` and rename `CommandResult.warnings` to `diagnostics: Vec<Diagnostic>`. Update all command engines to construct `diagnostics`. Replace warning string pushes with constructors. For disabled limits, use `Diagnostic::warning(WarningDiagnosticCode::LimitDisabled, "limit disabled: --max=unlimited")` or the existing `--max-depth` message. For truncation, use `WarningDiagnosticCode::OutputTruncated` and preserve the existing message text `truncated output to N entries (use --max to increase limit)`. For empty results, use `WarningDiagnosticCode::EmptyResult` with command-specific stable messages. Keep the existing `change` message `no signal changes found in selected time range`. Use `no scopes found` for `scope`, `no signals found in selected scope` for `signal`, `no property matches found in selected time range` for `property`, `no docs topics matched query` for `docs search`, and `no docs topics available` for `docs topics` if the catalog ever becomes empty. Add the empty-result diagnostic after filtering/truncation and before returning the result, so both human and JSON modes receive the same diagnostic list. Do not add empty-result diagnostics for `info`, valid `value`, `schema`, `docs show`, `docs export`, or `skill`.

Update `src/output.rs` next. Rename `OutputEnvelope::with_warnings` to a constructor such as `OutputEnvelope::new` or `with_diagnostics`, replace the `warnings` field with `diagnostics`, and update `render_json` to serialize `CommandResult.diagnostics`. Replace `emit_human_warnings` with `emit_human_diagnostics`. Its exact output must be `info: <message>` for info diagnostics, `warning[WPK-W0002]: <message>` for warning diagnostics, and `error[WPK-E0001]: <message>` for non-fatal error diagnostics. Because warning and error constructors require a code, the renderer should not silently invent or omit a code. Keep `exit_code()` unchanged: file errors exit `2`, all other categories exit `1`. Do not change `normalize_clap_error` in `src/cli/mod.rs` to search for `fatal:` because it parses clap's own rendered error text, which still begins with `error:` before being wrapped in `WavepeekError::Args`. Update top-level help text and any docs/dev examples that describe process-level error format so they say `fatal: <category>: <message>`.

The diagnostic definition must be an object with `additionalProperties: false`, required `kind` and `message`, `kind` enum values `info`, `warning`, and `error`, `message` as a string, and `code` as a string with pattern `^WPK-[WE][0-9]{4}$`. Add conditional schema rules so `warning` and `error` diagnostics require `code`, while `info` diagnostics must not contain `code`. ## Validation and Acceptance

The implementation is accepted when these behaviors are observable from the repository root.

For a normal successful JSON command:

    cargo run --quiet -- info --waves tests/fixtures/hand/m2_core.vcd --json

The output is one JSON object. It contains `$schema`, `command: "info"`, `data`, and `diagnostics: []`. It does not contain `warnings`.

For a truncated JSON command:

    cargo run --quiet -- scope --waves tests/fixtures/hand/m2_core.vcd --max 1 --json

The output is a successful JSON envelope with `diagnostics[0].kind == "warning"`, `diagnostics[0].code == "WPK-W0002"`, and a message beginning `truncated output to 1 entries`.

For a disabled-limit command in human mode:

    cargo run --quiet -- scope --waves tests/fixtures/hand/m2_core.vcd --max unlimited >tmp/stdout.txt 2>tmp/stderr.txt

The process exits `0`. `tmp/stderr.txt` contains `warning[WPK-W0001]: limit disabled: --max=unlimited`. Stdout contains the command payload.

For an empty successful list/search result:

## Idempotence and Recovery

All edits are ordinary source, test, and documentation changes. They can be repeated safely. If schema editing goes wrong, restore only the current-major artifact with `git checkout -- schema/wavepeek_v1.json` and reapply the planned diagnostic changes; do not modify `schema/wavepeek_v0.json`. Use repository-root `tmp/` for scratch stdout/stderr files and do not delete unrelated files there. If a broad search produces too much output, redirect it into `tmp/diagnostics-rg.log` and inspect only relevant stale public contract lines.

If `just check-schema` reports that the runtime schema differs from the artifact, remember that runtime schema output is embedded from `schema/wavepeek_v1.json`; rebuilds should pick up the edited file. Re-run `cargo clean` only as a last resort because it is expensive. Prefer `cargo run --quiet -- schema > tmp/schema.out` and `cmp schema/wavepeek_v1.json tmp/schema.out` to inspect the mismatch.

## Interfaces and Dependencies

Use existing dependencies only: `serde` for serialization, `serde_json` for rendering JSON, `thiserror` for process-level errors, `clap` for CLI parsing, and existing test crates already in `Cargo.toml`. No new crate is required.

At the end of the implementation, these Rust interfaces should exist or have equivalent names:

    // src/diagnostic.rs
    #[derive(Debug, Clone, Copy, PartialEq, Eq, serde::Serialize)]
    #[serde(rename_all = "lowercase")]
    pub enum DiagnosticKind {
        Info,
        Warning,
        Error,
    }

    #[derive(Debug, Clone, Copy, PartialEq, Eq)]
    pub enum WarningDiagnosticCode {
        LimitDisabled,
        OutputTruncated,
        EmptyResult,
    }

    impl WarningDiagnosticCode {
        pub const fn as_str(self) -> &'static str;
    }

    #[derive(Debug, Clone, Copy, PartialEq, Eq)]
    pub enum ErrorDiagnosticCode {}

    impl ErrorDiagnosticCode {
        pub const fn as_str(self) -> &'static str;
    }

    #[derive(Debug, Clone, PartialEq, Eq, serde::Serialize)]
    pub struct Diagnostic {
        kind: DiagnosticKind,
        #[serde(skip_serializing_if = "Option::is_none")]
        code: Option<&'static str>,
        message: String,
    }

    impl Diagnostic {
        pub fn info(message: impl Into<String>) -> Self;
        pub fn warning(code: WarningDiagnosticCode, message: impl Into<String>) -> Self;
        pub fn error(code: ErrorDiagnosticCode, message: impl Into<String>) -> Self;
        pub const fn kind(&self) -> DiagnosticKind;
        pub const fn code(&self) -> Option<&'static str>;
        pub fn message(&self) -> &str;
    }

Define `ErrorDiagnosticCode` as a strict reserved type rather than accepting arbitrary strings. A zero-variant enum is acceptable for the initial implementation because no production code emits non-fatal error diagnostics yet; its `as_str(self)` method can use an empty `match self {}` until the first real `WPK-E####` code is added. If tests need to exercise `error[WPK-E0001]: ...` rendering before a production code exists, add a private `#[cfg(test)]` constructor that validates the `WPK-E####` pattern instead of exposing arbitrary production codes.

`src/engine/mod.rs` should expose:

    pub struct CommandResult {

        pub json: bool,

        pub data: CommandData,
        pub diagnostics: Vec<crate::diagnostic::Diagnostic>,
